# Техническое задание (обновлённое) на разработку Telegram-бота «Новостной агрегатор с модерацией»

## 1. Введение
### 1.1. Назначение
Разработать автоматизированную систему сбора новостей из множества источников (сайты, Telegram-каналы, чаты, блоги), обработки их через AI для приведения к **заданному формату** и **в шуточном стиле**, с последующей модерацией администраторами через Telegram и публикацией в целевой канал. Система должна включать веб-интерфейс для гибкой настройки всех параметров работы.

### 1.2. Цели
- Автоматизация сбора и подготовки новостного контента.
- Сокращение времени ручной обработки.
- Исключение дублирования новостей.
- Обеспечение удобного интерфейса для администрирования и контроля качества.

## 2. Функциональные требования

### 2.1. Модуль сбора (парсинга)
- **Источники**: поддержка RSS/Atom лент, HTML-страниц (с использованием CSS-селекторов или XPath), Telegram-каналов/чатов (через клиент API), блогов (WordPress, Medium и др.).
- **Периодичность**: автоматический запуск каждые 30–60 минут (настраивается в веб-интерфейсе).
- **Извлечение данных**: заголовок, текст, дата публикации, автор (если есть), ссылка на оригинал, медиафайлы (изображения, видео) — скачивание или сохранение URL.
- **Фильтрация**: настраиваемые ключевые слова, исключение стоп-слов, минимальная длина текста.
- **Обработка ошибок**: повторные попытки при недоступности источника, логирование проблем.

### 2.2. Модуль обработки AI (с изменённым форматом вывода)
- **Интеграция**: подключение к внешнему AI API (например, OpenAI GPT, Claude API или локальная модель).
- **Формат ответа от ИИ**:  
  - ИИ **обязан** выдавать структурированный ответ **строго по шаблону**, который задаётся администратором в веб-интерфейсе.  
  - В системе предусмотрен **пример формата** (доступен в панели и в боте для ознакомления). Пример может выглядеть так:  
    ```json
    {
      "title": "Короткий заголовок (до 80 символов)",
      "intro": "Забавное вступление с юмором, 1–2 предложения",
      "body": "Основная часть с ироничными комментариями, до 500 символов",
      "comment": "Смешная подпись или забавный факт (опционально)",
      "hashtags": "#хэштеги #через #пробел"
    }
Стиль: все тексты (заголовок, вступление, основная часть, комментарий) должны быть в шуточном, лёгком, неформальном тоне – с уместными ироничными замечаниями, игрой слов, сленгом, чтобы посты напоминали популярные новостные паблики.

Администратор может редактировать промпт и пример формата в любое время через веб-интерфейс; изменения применяются сразу к новым постам.

Задача AI: преобразовать «сырой» текст новости в структурированный пост по заданному шаблону (JSON).

Сохранение оригинального текста и обработанного варианта в БД.

2.3. Модуль модерации (Telegram-бот)
Уведомления: после обработки AI, готовый пост отправляется в личные сообщения одному или нескольким администраторам (список настраивается).

Действия администратора (в интерфейсе бота через inline-кнопки):

«Доработать» – открывается поле для ручного редактирования текста (или отправка нового текста в ответ). После редактирования пост снова попадает в модерацию или отправляется на повторную AI-обработку.
«Опубликовать» – пост сразу отправляется в целевой Telegram-канал (с поддержкой форматирования, медиа, кнопок).
«Отказаться» – пост архивируется с пометкой «отклонён», не отправляется в канал.
Комментарии администратора: возможность оставить заметку к посту (сохраняется в БД).

2.4. Модуль публикации
Поддержка отправки текста с форматированием (Markdown/HTML), вложений (фото, видео, документы) и интерактивных кнопок.

Возможность отложенной публикации (задать время).

Логирование всех опубликованных постов.

2.5. Веб-интерфейс администрирования
Доступен по логину/паролю (или через OAuth). Основные разделы:

2.5.1. Панель управления
Статистика: количество обработанных постов за день, количество опубликованных, процент отклонения, состояние парсинга.

Графики активности.

2.5.2. Управление источниками
Добавление/редактирование/удаление источника:

Тип (RSS, HTML, Telegram, Blog).

URL/ID канала.

Параметры парсинга (CSS-селекторы, частота опроса).

Фильтры (ключевые слова, исключения).

Теги/категории для автоматической маркировки.

Активен/неактивен.

Ручной запуск проверки источника.

2.5.3. Управление модераторами
Добавление/удаление Telegram ID администраторов.

Назначение прав (только чтение, модерация, полный доступ).

2.5.4. Настройки AI (включая редактирование формата вывода)
Выбор модели API, ввод ключа доступа.

Редактирование промпта – текста запроса к AI (можно включить описание стиля и требуемого JSON-формата).

Редактирование примера формата – эталонного JSON, по которому ИИ должен строить ответ. В панели и в боте доступен предустановленный пример (забавный, неформальный).

Параметры температуры, максимальной длины ответа.

Тестовый режим для проверки качества обработки (отправка пробной новости и просмотр результата).

2.5.5. Управление публикациями
Настройка целевого канала (chat_id).

Шаблоны оформления постов (заголовок, подпись, хештеги).

Отложенные посты (список, редактирование, отмена).

2.5.6. Журнал событий
Логи всех действий: парсинг, обработка AI, модерация, публикация, ошибки.

Фильтры по дате, типу, пользователю.

2.5.7. Системные настройки
Периодичность парсинга (глобально).

Ограничение на количество постов за цикл.

Настройки уведомлений.

Резервное копирование БД.

2.6. Защита от дублей
Для каждого поста вычисляется хеш (на основе заголовка + даты + ссылки или текста) и сохраняется в БД.

Перед добавлением нового поста проверяется наличие такого хеша за последние N дней (настраивается).

При обнаружении дубликата – пост игнорируется, но информация логируется.

2.7. Работа с медиафайлами
Скачивание изображений/видео из источников (или использование прямых ссылок).

Сжатие/оптимизация изображений (опционально).

Загрузка медиа в Telegram при публикации (поддержка фото, видео, GIF, документов).

Медиа сохраняются в файловом хранилище (локальная папка или облачный S3) с привязкой к посту.

3. Технические требования
3.1. Стек технологий (рекомендуемый)
Backend: Python 3.10+ (FastAPI или Django) для веб-интерфейса и API.

Telegram-бот: aiogram или python-telegram-bot.

Парсинг: BeautifulSoup4, Selenium (для JavaScript-сайтов), feedparser, Telethon (для Telegram API).

База данных: PostgreSQL (основная) + Redis (кэш и очереди задач).

Очереди задач: Celery + Redis (для фоновых задач парсинга и AI-обработки).

Файловое хранилище: локальная файловая система или Amazon S3/MinIO.

Веб-интерфейс: React/Vue.js (SPA) или Django Templates (для простоты) + Bootstrap.

Деплой: обязательно использование Docker и docker-compose для всех сервисов (app, bot, celery-worker, redis, postgres, nginx). Вся конфигурация должна быть описана в репозитории.

3.2. Архитектура
Монолитное приложение с выделенными модулями (или микросервисы: парсер, AI-обработчик, бот, веб-интерфейс). Для простоты – монолит с Celery.

Асинхронное выполнение длительных задач (парсинг, AI) через очереди, чтобы не блокировать веб-интерфейс и бота.

Кэширование часто запрашиваемых данных (список источников, настройки).

3.3. Интеграции
Telegram Bot API: для отправки сообщений, медиа, инлайн-кнопок.

Telegram Client API (Telethon): для парсинга публичных каналов/чатов (при необходимости).

AI API: REST-запросы к OpenAI / Anthropic / YandexGPT / локальной модели (поддержка конфигурируемого эндпоинта).

Веб-интерфейс: общается с бекендом через REST API (JWT-аутентификация).

3.4. Требования к производительности
Обработка не менее 100 источников с суммарным количеством постов до 500 за один цикл.

Время ответа веб-интерфейса – не более 2 секунд.

Отправка уведомлений админам – не более 5 секунд.

Работа с AI – с учётом задержек API, использование асинхронных вызовов.

3.5. Безопасность
Хранение API-ключей и паролей в переменных окружения (не в коде).

HTTPS для веб-интерфейса.

Ограничение доступа по IP (опционально).

Регулярное обновление зависимостей.

Логирование без чувствительных данных.

3.6. Деплой и сопровождение
Весь проект должен быть упакован в Docker-контейнеры с использованием docker-compose. В репозитории должны присутствовать:

Dockerfile для каждого сервиса (backend, bot, celery-worker, nginx, и т.д.).

docker-compose.yml с описанием всех сервисов, сетей и томов.

Пример .env файла со всеми необходимыми переменными.

Инструкция по выгрузке на хост (VPS):

Подготовка сервера (установка Docker, docker-compose, Git).
Клонирование репозитория, настройка переменных окружения (скопировать .env.example в .env и заполнить).
Запуск docker-compose up -d и первоначальная инициализация БД (миграции) – должна быть автоматизирована (например, entrypoint).
Настройка обратного прокси (Nginx) для веб-интерфейса и HTTPS (Let's Encrypt) – опционально, но желательно.
Рекомендации по мониторингу (проверка работающих контейнеров, просмотр логов через docker-compose logs).

4.1. Telegram-бот (интерфейс для модераторов)

Адаптивный дизайн (работает на ПК и планшетах).

Авторизация по логину/паролю (сессия, двухфакторность опционально).

Основные страницы:

Источники – таблица со статусами, поиск, фильтры.

5. База данных (основные сущности)
Source: id, name, type (rss/html/telegram/blog), url, config (JSON), filters (JSON), active, last_checked, created_at.

Post: id, source_id, original_title, original_text, original_url, hash, raw_media (JSON), status (pending, processed, moderated, published, rejected), ai_processed_text (JSON с полями title, intro, body, comment, hashtags), ai_category, ai_tags, moderation_comment, created_at, published_at.

Media: id, post_id, file_path, file_type, size, url (если внешняя).

Admin: id, telegram_id, username, role (viewer, moderator, admin), added_at.

Settings: key, value (JSON или текст) – здесь хранится промпт, пример формата, другие параметры.

Log: id, timestamp, level, message, module, details (JSON).

6. Логика обработки потока
Парсинг (задача Celery, выполняется по расписанию):

Для каждого активного источника получаем новые посты (с момента последнего запуска).

Для каждого поста вычисляем хеш и проверяем дубликат → если дубль – пропускаем.

Сохраняем сырой пост в БД со статусом processed (чтобы показать, что он получен).

AI-обработка (отдельная задача или цепочка):

Берём посты со статусом processed, отправляем в AI API с текущим промптом и требованием вернуть JSON по примеру формата.

Парсим JSON, проверяем соответствие схеме. Если ошибка – повторяем запрос или помечаем пост как ошибочный.

Сохраняем обработанный JSON в поле ai_processed_text, обновляем статус на moderated.

Модерация:

Система отправляет сообщение администраторам (всем, кто активен) с постом (собранным из полей JSON) и кнопками.

Администратор выбирает действие.

Если «Доработать» – ждём ввод нового текста, заменяем AI-версию (или отдельные поля), снова отправляем админу (или повторно отправляем в AI с изменённым промптом).

Если «Опубликовать» – отправляем в канал, меняем статус на published.

Если «Отказаться» – статус rejected, архивируем.

Публикация – может быть немедленной или отложенной (если задано время).

11. Критерии приёмки

Проект успешно собирается в Docker-контейнеры и запускается на тестовом сервере по инструкции.
